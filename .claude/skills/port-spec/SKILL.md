# Port Spec — 기능 이식 스펙 추출

## 핵심 원칙

1. **자기완결**: 스펙을 받는 세션은 원본 레포에 접근할 수 없다. 구현에 필요한 값(스키마 컬럼, payload, 에러 문구, 검증 규칙, UX 문구)은 전부 스펙 본문에 인라인한다. 원본 file:line·커밋 해시는 부록 대응표(검증용)에만 둔다.
2. **스펙이 결정한다**: 설계 지점마다 선택지를 나열하고 "대상 세션이 관례를 보고 결정"이라고 유보하는 것은 스펙 실패다. 대상 레포 조사를 근거로 스펙 작성자가 확정하고 근거 1줄을 남긴다. 조사로도 확인 불가한 항목만 Open Questions에 남긴다.
3. **코드가 진실**: 설계 문서·기억은 힌트일 뿐, 스키마·검증·문구는 현재 코드에서 실측한다. 파일 포맷 같은 산출물 계약은 실제 실행 결과가 정답이다.

## 프로세스

1. **스코프 고정** — 기능의 어느 부분인지 확정(예: "서버 설정만"). 모호하면 진행 전에 1회 질문. 제외 목록을 스펙에 명시.
2. **병렬 조사** (서브에이전트/Workflow 병렬 파견):
   - 원본: 기존 설계 문서 + 실코드 — 데이터 모델, API, UI 흐름, 검증·엣지케이스·함정, 테스트 시나리오
   - 대상: 스택, 디렉토리·API·마이그레이션 관례, 권한 모델, UI 컴포넌트 관례, 기존 유사/충돌 코드 — 통합 지점과 난관 발견이 목적
3. **번역 설계** — 원본 개념을 대상 스택 용어로 재설계. 원본 스택 사정으로 생긴 우회(예: DB 제약 회피, 파일 실체화)는 대상에서 불필요하면 버리고 버린 사실을 부록에 기록.
4. **작성** — 아래 출력 계약 순서대로.
5. **적대 검증** — 별도 서브에이전트에 스펙 파일만 주고: "이 스펙만으로 질문 없이 구현 가능한가? 스펙이 전제한 대상 레포의 파일·헬퍼·관례가 실존하는가?" 결함 보강 후 전달.

## 출력 계약 (스펙 문서 섹션 순서)

1. 목적·스코프 — 제외 항목 명시
2. 개념 모델 — 용어 정의 표
3. 데이터 모델 — 대상 DB 기준, 마이그레이션 초안 포함
4. API/액션 계약 — 시그니처, payload/응답 타입, 에러 문구 원문
5. 핵심 통합 지점 — 대상 기존 코드와 만나는 가장 위험한 부분을 별도 섹션으로 승격
6. UI/UX — 화면 구조, 상태 전이, 사용자 문구
7. 검증·에러 규칙 표
8. 보안 체크리스트 — 시크릿 저장·마스킹·권한 게이트
9. 구현 순서 — 단계별 테스트 게이트 포함
10. 수용 기준 — 수동 검증 시나리오 번호 목록
11. 부록 — 원본 대응표(원본 file ↔ 스펙 섹션), 의도적으로 버린 원본 요소

## 흔한 실수

| 실수 | 결과 |
|---|---|
| 선택지 나열 후 결정 유보 | 대상 세션이 멈춰서 되물음 — "바로 작업" 실패 |
| 원본 경로·커밋을 구현 지시에 사용 | 대상 세션은 원본 접근 불가 |
| 설계 문서만 믿고 실코드 미확인 | 낡은 스펙 |
| 원본 스택 사정을 그대로 이식 | 불필요한 복잡도 복제 |
| 대상 레포 파일·헬퍼 실존 미검증 | 스펙이 유령 코드 참조 |
| 시크릿 취급 규칙 누락 | 키 노출·마스킹 빠짐 |
