# 또박또박 윈도우 서버 운영 가이드 (런북)

배포가 끝난 윈도우 서버(WSL2)를 일상적으로 운영·관리하는 방법입니다.
설치 절차는 [`윈도우-서버-WSL2-배포가이드.md`](윈도우-서버-WSL2-배포가이드.md), 접속 정보(IP·계정·비밀번호)는 git 미추적 `_local/서버-접속정보.md`를 참고하세요.

## 구성 한눈에 보기

| 항목 | 값 |
|---|---|
| 호스트 | Windows 11 Pro + WSL2 Ubuntu (NVIDIA RTX 5000 Ada, VRAM 32GB) |
| 소스 위치 | WSL Ubuntu `/home/ddobak/ddobakddobak` (유저 `ddobak`) |
| 웹 접속 | `https://<서버IP>` (자체서명 — 브라우저 경고 1회 수락) |
| 서비스 | `ddobak-backend`(Rails :13323) · `ddobak-stt`(sidecar :13324) · `nginx`(:443) |
| DB | `backend/storage/production.sqlite3` (+ `_queue`/`_cable`/`_cache`) |
| 오디오 | `backend/storage/audio/` |
| 설정 | `~/ddobakddobak/settings.yaml` |
| STT 엔진 | 실시간 = `auto`(CUDA→Qwen3-ASR GPU) / 배치 = `faster_whisper`(Whisper large-v3-turbo GPU) |
| 화자분리 | speakrs(Rust) — Linux 빌드본 `/home/ddobak/bin/speakrs-cli` (`SPEAKRS_BIN` env로 지정) |
| 모델 캐시 | `~/.cache/huggingface/hub` |
| 로그 | systemd journald (`journalctl -u <서비스>`) |

## 접속

```sh
ssh ddobak-win        # 맥에서 → 바로 WSL Ubuntu 셸 (키 인증)
ssh ddobak-win-ps     # 맥에서 → Windows PowerShell (포트·방화벽·WSL 관리용)
```

- 맥 `~/.ssh/config`에 위 두 항목이 등록되어 있다 (배포가이드 2단계).
- RDP는 GUI가 꼭 필요한 예외 상황(드라이버 설치 등)에만 사용.

## 서비스 관리

```sh
# 상태 확인 (셋 다 active여야 정상)
systemctl status ddobak-backend ddobak-stt nginx

# 재시작
sudo systemctl restart ddobak-backend   # Rails
sudo systemctl restart ddobak-stt       # STT sidecar (재시작 시 모델 재로드 ~1분)
sudo systemctl reload nginx             # nginx 설정 반영

# 실시간 로그
journalctl -u ddobak-backend -f
journalctl -u ddobak-stt -f
```

⚠️ `ddobak-stt` 재시작은 진행 중인 전사·화자분리 작업을 끊는다. 작업 없는 때에 할 것.

## 헬스 체크

### 화자분리 상태 확인

화자분리는 별도 바이너리 `speakrs`(Rust)를 subprocess로 호출한다. 이 서버는 Linux 빌드본을 `/home/ddobak/bin/speakrs-cli`에 두고 systemd 유닛의 `SPEAKRS_BIN` 환경변수로 지정했다 (repo의 `sidecar/bin/speakrs-cli`는 맥 arm64 바이너리라 덮어쓰지 않기 위함). 바이너리가 없으면 **전사는 정상 진행되고 화자 라벨만 안 붙는다**(전사 실패 아님).

```sh
# 1) 바이너리 존재·실행권한 확인 (Linux speakrs 빌드 결과물)
ls -l /home/ddobak/bin/speakrs-cli
#   실행권한(x)이 있는 ELF 파일 → 활성 / 파일 없음 → 재빌드 필요 (배포가이드 진행 기록 참고)

# 2) 파일 전사 1건 후 로그로 실동작 확인
journalctl -u ddobak-stt | grep -iE "화자|speakrs" | tail
#   "배치 화자 분리 완료 (speakrs)"        → 화자분리 동작 중
#   "화자 분리 엔진 사용 불가(speakrs 미설치)" → 아직 미빌드 (라벨 없이 전사만 진행)
```

- 사용자 환경변수 `SPEAKRS_BIN`이 설정돼 있으면 그 경로가 우선한다.
- 활성 여부의 실질 게이트는 **바이너리 존재**다. `/health`·`/settings/stt-engine` 응답에는 화자분리 상태가 들어 있지 않으므로 위 방법으로 확인한다.

## 코드 업데이트 (배포)

```sh
ssh ddobak-win
cd ~/ddobakddobak
git pull

# 변경 범위에 따라:
sudo systemctl restart ddobak-backend             # backend/ 변경 시
sudo systemctl restart ddobak-stt                 # sidecar/ 변경 시
cd frontend && npm install && npx vite build      # frontend/ 변경 시 (dist 재생성, 서비스 재시작 불필요)

# backend gem 변경 시
cd ~/ddobakddobak/backend && bundle install
RAILS_ENV=production bin/rails db:migrate         # 마이그레이션 있을 때
sudo systemctl restart ddobak-backend

# sidecar 의존성 변경 시
cd ~/ddobakddobak/sidecar && uv sync --extra cuda
sudo systemctl restart ddobak-stt
```

> **기업망 CA (SSL 가로채기)**: 이 서버는 프록시가 TLS를 가로채는 기업망에 있다. 기업 CA(CN=dongkuk)가 시스템에 등록돼 있고, `SSL_CERT_FILE`·`REQUESTS_CA_BUNDLE`(Python)·`NODE_EXTRA_CA_CERTS`(Node) 환경변수가 `~/.bashrc`에 설정돼 있다. `git`/`bundle`/`npm`/`uv`에서 `certificate verify failed`가 나면 배포가이드의 "문제 해결" 표(기업망 SSL 항목)를 따른다.

## 개선(개발) 작업 시작하기 — 맥에서

원칙: **개발·커밋·푸시는 맥에서, 서버는 pull-only.** 서버엔 git push 인증이 없어 서버에서 커밋하면 고아 커밋이 된다(실제 사고 있었음).

**1) 맥에서 시작**

```sh
cd ~/project/ddobakddobak
claude          # 새 세션이면 "윈도우 서버 작업 이어서"라고 하면 맥락 복원됨
```

**2) 수정 → 커밋 → `git push origin main`**

**3) 서버 반영**

```sh
ssh ddobak-win                      # WSL Ubuntu 직행
cd ~/ddobakddobak && git pull --ff-only
```

**4) 변경 영역별 반영 절차**

- **backend 변경**: 마이그레이션 있으면 `cd backend && bin/rails db:migrate RAILS_ENV=production` → `sudo systemctl restart ddobak-backend`
- **sidecar 변경**: `cd sidecar && uv sync --extra cuda` → `sudo systemctl restart ddobak-stt`
- **frontend 변경**: `cd frontend && npm run build` → dist가 재생성되므로 `cp ~/ddobak.crt dist/` 재복사 (nginx는 정적 서빙이라 재시작 불필요)

**5) 확인**

```sh
systemctl is-active ddobak-backend ddobak-stt nginx
# 이상 시:
journalctl -u <서비스> -n 50
```

> **참고**: 서버 WSL엔 Claude Code 미설치 — 서버 안 직접 개발은 현재 워크플로가 아니다. 위처럼 맥에서 개발하고 서버는 pull만 한다.

## 설정 변경 (settings.yaml)

`~/ddobakddobak/settings.yaml` 수정 후 `sudo systemctl restart ddobak-stt` (sidecar가 시작 시점에 읽음).

```yaml
stt:
  file_engine: faster_whisper   # 배치(파일) 전사 엔진. GPU Whisper large-v3-turbo
  engine: auto                  # 실시간 엔진. auto=CUDA에서 Qwen3-ASR 자동선택
```

⚠️ 관리 UI에서 STT 설정을 저장하면 이 값을 덮어쓸 수 있다. 저장 후 `file_engine`이 `whisper_cpp`(CPU, 1배속)로 바뀌지 않았는지 확인.
⚠️ `settings.yaml`에는 HF 토큰·LLM API 키가 들어간다. **값을 문서·채팅·커밋에 붙여넣지 말 것** (git 미추적 대상).

## 백업

```sh
# DB + 오디오가 전부. 이 두 경로만 백업하면 된다
tar czf ~/ddobak-backup-$(date +%Y%m%d).tgz \
  -C ~/ddobakddobak/backend/storage production.sqlite3 audio
```

- 앱 자체의 프로젝트/회의 export 기능(관리 UI)도 병용 가능.
- 참고: 위 `tar`는 가동 중인 SQLite(WAL 모드)를 그대로 뜨므로 드물게 일부 시점이 어긋난 사본이 될 수 있다. 정합성이 중요하면 백업 직전에 `cd ~/ddobakddobak/backend/storage && sqlite3 production.sqlite3 ".backup ~/production.bak.sqlite3"`로 일관 사본을 먼저 만든다.

## 디스크 용량 관리

무한정 커지는 것은 **오디오와 DB**다. 모델 캐시는 로드하는 모델 몇 개로 크기가 사실상 고정(bounded)이라 증가 원인이 아니다.

```sh
df -h ~                                              # WSL(ext4) 전체 여유 공간
du -sh ~/ddobakddobak/backend/storage/audio          # 오디오 누적 (성장 1순위)
du -sh ~/ddobakddobak/backend/storage/*.sqlite3      # DB 파일들
du -sh ~/.cache/huggingface/hub                      # 모델 캐시 (거의 고정)
```

- 오디오는 회의를 지우면 앱이 함께 정리한다. 백업 후 정리하는 것이 안전.
- ⚠️ WSL 안에서 파일을 지워도 그 공간이 **자동으로 Windows 호스트에 반환되지 않는다**(WSL 가상 디스크는 자동 축소 안 함). 호스트 디스크를 실제로 되찾는 작업은 Windows 쪽(`ddobak-win-ps`)에서 수행하는 별도 절차이며, 디스크가 실제로 임박했을 때만 신중히 진행한다.

## 로그 관리 (journald)

세 서비스 로그는 systemd journald에 쌓인다. 저널이 커지면 정리한다.

```sh
journalctl --disk-usage                    # 저널 총 사용량
sudo journalctl --vacuum-time=14d          # 14일 이전 로그 삭제
sudo journalctl --vacuum-size=500M         # 500MB로 상한 정리
```

- 영구 상한은 `/etc/systemd/journald.conf`의 `SystemMaxUse=`(예: `500M`) 지정 후 `sudo systemctl restart systemd-journald`.

## 인증서 (nginx 자체서명)

웹은 배포 시 만든 **자체서명 인증서**(CN=서버IP, 유효기간 약 10년)를 쓴다. 브라우저 경고 1회 수락 방식이라 **평상시 갱신이 필요 없다** — 아래는 참고용이다.

```sh
# 현재 제공 중인 인증서 만료일 확인 (파일 경로 몰라도 됨)
echo | openssl s_client -connect localhost:443 2>/dev/null | openssl x509 -noout -enddate
```

- 서버 IP가 바뀌거나(CN 불일치) 10년 뒤 만료가 임박할 때만 자체서명 인증서를 재발급하고 `sudo systemctl reload nginx`로 반영한다.

### 클라이언트에서 브라우저 경고 없애기 (선택)

자체서명이라 기기·브라우저마다 "연결이 비공개로 설정되어 있지 않습니다" 경고가 뜬다. "고급 → 이동" 1회 수락으로 충분하지만, 경고 자체를 없애려면 접속하는 **각 기기의 신뢰 저장소에 인증서를 등록**한다 (서버 쪽 조치로는 불가 — 공인 도메인이 없어 Let's Encrypt 사용 불가).

인증서는 `https://<서버IP>/ddobak.crt`로 내려받을 수 있다 (frontend `dist/`에 복사해 둠 — `vite build`로 dist 재생성 시 사라지므로 `~/ddobak.crt` 원본을 다시 복사).

**Windows (Chrome·Edge)** — 관리자 cmd에서 한 줄:

```bat
curl.exe -k https://<서버IP>/ddobak.crt -o "%TEMP%\ddobak.crt" && certutil -addstore Root "%TEMP%\ddobak.crt"
```

또는 GUI: `.crt` 더블클릭 → 인증서 설치 → **로컬 컴퓨터** → **신뢰할 수 있는 루트 인증 기관**. 등록 후 브라우저 완전 재시작. 해당 PC의 모든 사용자에게 적용된다.

**macOS (Chrome·Edge)**:

```sh
curl -sk https://<서버IP>/ddobak.crt -o ~/Downloads/ddobak.crt
security add-trusted-cert -r trustRoot -k ~/Library/Keychains/login.keychain-db ~/Downloads/ddobak.crt
```

- ⚠️ Safari/iOS는 유효기간 825일 초과 인증서를 신뢰 등록해도 거부할 수 있다(Apple 정책, 이 인증서는 10년). Chrome·Edge는 제한 없음.
- Firefox는 자체 저장소를 쓰므로 별도 등록 필요 (설정 → 인증서 → 가져오기).
- 등록 완료 기기(2026-07-14): 맥(login 키체인), 윈도우 서버 본체(LocalMachine Root).

## WSL 관리

WSL 자체의 시작/종료는 Windows 쪽에서 다룬다 (`ssh ddobak-win-ps`).

```sh
ssh ddobak-win-ps      # Windows PowerShell

wsl -l -v              # distro·버전 확인 (Ubuntu / VERSION 2)
wsl --status           # 기본 distro·WSL 버전

# WSL 완전 종료 (모든 서비스·tmux가 죽는다) — .wslconfig 변경 반영 등에만 사용
wsl --shutdown
```

⚠️ `wsl --shutdown` 이후에는 keep-alive 작업을 **수동으로 다시 실행**해야 WSL이 다시 뜨고 systemd 서비스가 자동 기동된다. 안 하면 WSL이 내려간 채로 남는다.

```sh
schtasks /run /tn WSL-AutoStart
```

- 재부팅(아래)은 이 태스크가 자동으로 도므로 수동 실행이 필요 없다. `wsl --shutdown`처럼 재부팅 없이 수동으로 내린 경우에만 위 명령이 필요하다.
- 네트워킹(mirrored)·유휴 종료(vmIdleTimeout) 설정은 Windows의 `%USERPROFILE%\.wslconfig`에서 바꾸며, 변경 반영에도 `wsl --shutdown`이 필요하다.

## 재부팅

그냥 재부팅하면 된다 — 전체 스택이 자동 복구된다 (검증 완료):
sshd 자동시작 → 작업 스케줄러 `WSL-AutoStart`(keep-alive) → systemd가 3서비스 기동.

```sh
ssh ddobak-win-ps "shutdown /r /t 0"
# 2~3분 후 재접속·헬스 체크
```

| 파일 전사가 0%에서 안 움직임 | ① sidecar health 확인(모델 로딩 중이면 대기) ② `journalctl -u ddobak-stt -f`로 진행 확인 ③ 큐 상태: 아래 "잡 큐 점검" |
| 전사가 너무 느림 (1배속) | `settings.yaml`의 `file_engine`이 `whisper_cpp`(CPU)로 바뀐 것. `faster_whisper`로 복구 후 sidecar 재시작 |
| 화자 라벨이 안 붙음 | speakrs 바이너리 미빌드/부재. "화자분리 상태 확인"으로 점검 (전사 자체는 정상) |
| GPU 부하가 안 보임 | Windows 작업관리자는 WSL CUDA를 못 본다. WSL 안 `nvidia-smi`로 확인 |
| 재부팅 후 WSL·서비스 안 뜸 | `ddobak-win-ps`에서 `schtasks /run /tn WSL-AutoStart` 실행. 상습이면 태스크 등록 상태 점검(배포가이드 3단계) |
| ls에 한글이 `\353...`으로 | 로케일 문제. 이미 `ko_KR.UTF-8` 설정됨 — 새 셸로 재접속 |
| 디스크 가득 참 | "디스크 용량 관리"(오디오·DB) + "로그 관리"(journald) 참고 |
| 웹 접속 안 됨 | `systemctl status nginx` → 방화벽(443) → `.wslconfig` mirrored 확인 |
| 로그인 404 | nginx `/auth/` 프록시 누락 여부 (`/etc/nginx/sites-available/ddobak`) |
| 회의가 `transcribing`에 고착 | 잡이 죽은 것. 아래 잡 큐 점검 후 UI에서 재전사 |

### 잡 큐 점검

```sh
cd ~/ddobakddobak/backend
RAILS_ENV=production bin/rails runner 'puts "ready=#{SolidQueue::ReadyExecution.count} claimed=#{SolidQueue::ClaimedExecution.count} failed=#{SolidQueue::FailedExecution.count}"'
# failed가 있으면:
RAILS_ENV=production bin/rails runner 'SolidQueue::FailedExecution.all.each { |f| puts "#{f.job.class_name}: #{f.error.to_s[0,120]}" }'
```

⚠️ 새 잡에 `queue_as :새큐이름`을 쓰면 반드시 `backend/config/queue.yml`의 워커에도 추가할 것 — 빠지면 production에서 잡이 영원히 `ready`로 대기한다 (개발환경은 async 어댑터라 재현 안 됨). 현재 등록된 큐: `real_time`, `summarization`, `file_transcription`, `default`, `card_extraction`, `solid_queue_recurring`(recurring.yml `command:` 태스크 기본 큐 — 빠지면 매시 cleanup이 영원히 대기).
