# Tauri 백그라운드/트레이/예약 자동시작 — 기기 검증 체크리스트

- 날짜: 2026-06-22
- 브랜치: `feat/tauri-background-tray`
- 대상 OS: macOS (Phase 1)

## 머리말

이 문서는 Tauri 백그라운드 실행·트레이/메뉴바·예약 백그라운드 자동시작 구현(Task 1~11)에 대한
**실기기 수동 E2E 검증** 체크리스트다.

cargo build / cargo test / vitest는 CI에서 통과 완료. 아래 항목들은 GUI 동작·시스템 슬립
동작·OS 알림·caffeinate 효과처럼 **헤드리스 테스트로 검증 불가능**한 것들이어서 실기기에서
직접 확인해야 한다.

```bash
cd /Users/jji/project/ddobakddobak/frontend
```

```bash
cd /Users/jji/project/ddobakddobak/frontend

# 빌드된 앱: src-tauri/target/release/bundle/macos/또박또박.app
```

```bash
curl -s http://127.0.0.1:13323/up        # "OK" 나오면 Rails 살아있음

pmset -g assertions                       # 슬립 차단 상태 확인
```

### Step 1: 트레이/닫기 동작 (Task 1 · 2 · 3)

**메뉴바 아이콘**

- [ ] 앱 실행 시 macOS 메뉴바 우측에 또박또박 아이콘이 표시된다.
- [ ] 아이콘에 마우스 오버 시 툴팁 "또박또박"이 나타난다.

**좌클릭 창 토글**

- [ ] 메뉴바 아이콘 좌클릭 → 창이 표시된다 (숨겨져 있었다면).
- [ ] 메뉴바 아이콘 좌클릭 → 창이 숨겨진다 (보이고 있었다면).

**트레이 메뉴**

- [ ] 메뉴바 아이콘 우클릭(또는 메뉴 오픈) → "열기" / "완전 종료" 항목이 보인다.
- [ ] "열기" 클릭 → 창이 표시되고 포커스가 잡힌다.
- [ ] "완전 종료" 클릭 → 앱·Rails·sidecar가 모두 종료된다.
  - 확인: `curl -s http://127.0.0.1:13323/up` 응답 없음, `pgrep caffeinate` = 0.

**빨간 X (닫기 버튼) → 닫기 모달**

- [ ] 빨간 X 클릭 → 모달("또박또박을 어떻게 할까요?")이 나타난다.
- [ ] 모달에 "백그라운드 유지" / "완전 종료" 버튼과 "다음부터 묻지 않기" 체크박스가 있다.

**[백그라운드 유지] 선택**

- [ ] "백그라운드 유지" 클릭 → 창이 숨겨진다.
- [ ] 메뉴바 아이콘이 여전히 남아있다.
- [ ] `curl -s http://127.0.0.1:13323/up` → "OK" (Rails 살아있음).
- [ ] `pgrep caffeinate` — 녹음 중이 아니라면 0, 녹음 중이라면 숫자.

**[완전 종료] 선택**

- [ ] "완전 종료" 클릭 → 앱·Rails·sidecar가 모두 종료된다.
  - 확인: `curl -s http://127.0.0.1:13323/up` 응답 없음.
- [ ] `pgrep caffeinate` = 0 (caffeinate 잔존 없음).

**"다음부터 묻지 않기" 기억**

- [ ] "다음부터 묻지 않기" 체크 후 "백그라운드 유지" 클릭.
- [ ] 다음 번 빨간 X 클릭 → 모달 없이 즉시 창 숨김.
- [ ] 앱 재실행 후 빨간 X → 모달 없이 즉시 창 숨김 (localStorage 유지 확인).
- [ ] 설정에서 선택 초기화 후 빨간 X → 다시 모달 표시 (선택 사항).

**cmd+Q** ⚠️ 기기 E2E 최우선 위험 항목

- [ ] cmd+Q → 모달 없이 앱·Rails·sidecar가 즉시 종료된다.
- [ ] `pgrep caffeinate` = 0.
  - **실패 모드**: cmd+Q 후 창만 사라지고 `pgrep caffeinate`가 살아있으면, CloseRequested
    인터셉터(또는 커스텀 메뉴)가 quit 이벤트를 삼키고 있는 것이다. 트레이 "완전 종료"는 별도
    경로이므로 그쪽이 정상 종료되는지 비교 확인한다.

### Step 2: 예약 백그라운드 자동시작 (Task 4 · 5 · 6 · 8 · 9)

- [ ] 예약 생성 후 창을 "백그라운드 유지"로 숨긴다.
- [ ] 예약 시각 도달 시 창이 자동으로 표시되고 포커스가 잡힌다.

  - 알림이 안 뜨면 시스템 환경설정 > 알림 > 또박또박 권한 확인.

**manual 모드 예약**

### Step 3: Power assertion / 슬립 차단 (Task 7)

**녹음 중 caffeinate 확인**

- [ ] 녹음 시작 → `pgrep caffeinate` → 프로세스 번호가 나타난다.
- [ ] 녹음 종료 → `pgrep caffeinate` → 0 (사라짐).
- [ ] 앱 완전 종료 후 → `pgrep caffeinate` → 0 (누수 없음).

**T-120s 선제 assertion (예약 임박)**

- [ ] 예약 시각 ~2분 전부터 `pmset -g assertions` 확인:
  ```bash
  pmset -g assertions
  # PreventUserIdleSystemSleep 항목에 "또박또박" 또는 "caffeinate" 표시 확인
  ```
- [ ] 트리거 전에 이미 `PreventUserIdleSystemSleep`이 활성화되어 있다.

**디스플레이 슬립 상태에서 예약 실행**

- [ ] 예약 생성 후 "백그라운드 유지"로 앱 숨김.
- [ ] 디스플레이만 끈다 (설정 > 디스플레이 > 잠자기: 1분으로 단축 후 대기, 또는 `pmset displaysleep 1`).
  - 화면 꺼짐 = 디스플레이 슬립, 시스템은 깨어있는 상태.
- [ ] 예약 시각 도달 → 창이 자동 표시되고 녹음이 시작된다.
- [ ] 검증 후 displaysleep 원래 값으로 복원.

**유휴 방치 (시스템 슬립 임박) 상태에서 예약 실행**

- [ ] 예약 생성, "백그라운드 유지"로 앱 숨김.
- [ ] 시스템 유휴슬립 시간을 짧게 설정 (`sudo pmset -a sleep 2` 또는 에너지 설정에서 2분).
- [ ] 유휴 대기 → `pmset -g assertions`로 PreventUserIdleSystemSleep 활성 확인.
- [ ] 시스템이 잠들지 않고 예약 시각에 창이 표시되고 녹음이 시작된다.
- [ ] 검증 후 슬립 설정 원복 (`sudo pmset -a sleep 10` 등).

**딥슬립 (뚜껑 닫음) — Phase 2 범위 밖**

- [ ] (참고용, 통과 기준 없음) 뚜껑을 닫고 딥슬립 진입 후 예약 시각 경과 → 앱이 트리거하지 못함.
- [ ] 뚜껑 열어 깨운 후 → 예약이 "놓친 예약" pending 상태로 남아있다 (삭제되지 않음).

## 알려진 미해결 / 결정 필요 항목

### ✅ (a) 데스크톱 manual 예약 자동시작 확인 다이얼로그 — 해결됨

데스크톱 `manual` 모드 예약 트리거 시 `confirmDialog('예약된 회의를 시작할까요?')`를 표시하도록
결정·구현 완료. "예" → 즉시 시작, "아니오" → 시작 안 함(녹음 중 알림도 없음). `auto` 모드는
확인 없이 즉시 자동 시작(불변).

### (b) 트레이 숨김 시 무음 자동완료 타이머 throttle

창이 숨겨진 상태(트레이로 hide)에서 WKWebView의 `setInterval`이 App Nap으로 인해
스로틀될 수 있다. 이 경우 무음 5분 카운터가 제대로 동작하지 않을 수 있다.

- 현재는 트리거 시 `window.show()`로 창을 먼저 표시하므로 녹음 중엔 창이 보이는 상태.
- 그러나 사용자가 녹음 중에 수동으로 창을 숨기면 타이머가 throttle될 수 있다.
- 이 경우 자동완료가 동작하지 않는 fail-safe(미종료) 동작을 함.
- Phase 1에서는 허용 가능한 한계로 문서화. 창 표시 상태에서 검증 권장.

### (c) 딥슬립 (뚜껑 닫음) = Phase 2 범위 밖

뚜껑을 닫으면 macOS는 hibernation(딥슬립)에 진입하며, `caffeinate -is`로는 차단 불가.
이 케이스는 Phase 2(SMAppService root 헬퍼 + `pmset schedule wake` + 공증)에서 다룬다.
Phase 1에서는 딥슬립 중 예약이 실행되지 않는 것이 의도된 동작이며, 놓친 예약은 pending으로
유지된다(삭제되지 않음).
