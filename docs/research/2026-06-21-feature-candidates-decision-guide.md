## 0. 포지셔닝 — 우리가 누구인가 (선택의 기준선)

또박또박은 **한국어·자체호스팅(self-hosted) 팀 회의록**이다. Rails + SQLite / React / Tauri + Android. altalt(개인·온디바이스)이나 클로바노트(클라우드·개인)와 달리, **공유·라이브 호스트 이양·폴더·팀**을 갖춘 협업형이고, 전 코퍼스가 로컬에 있다 → 클라우드 경쟁사가 못 내는 **프라이버시·데이터 소유** 우위.

이 정체성이 후보 선택의 기준이다. "클라우드 경쟁사가 SaaS로 파는 것"을 따라가기보다, **자체호스팅이기에 더 잘 맞는 것**(웹훅·API·거버넌스)과 **데이터를 이미 갖고 있어 거의 공짜인 것**(발언 분석·다포맷)을 먼저 본다.

**핵심 진단(두 조사 공통):** 현 기능은 강하지만 *사후(post)·배치·텍스트 출력*에 편중돼 있다. 가장 큰 빈틈은 **라이브(회의 중) 가치** — 가장 어려운 전제(라이브 페이지 + ActionCable + per-user 챗 RAG)를 이미 깔아 놓고도 회의 중에는 거의 안 쓰고 있다.

## 3. Tier 1 — 저비용 즉효 (강력 추천, 먼저 하라)

데이터·인프라를 **이미 갖고 있어** 새로 캡처하거나 외부 비용을 들일 게 없는 것들. 한 스프린트 안에 체감 효과가 가장 크다.

### ① 화자별 발언 점유율·분석 ★★★ · 공수 S

**무엇인가.** 회의가 끝나면 "누가 얼마나 말했나"를 보여 준다. 화자별 발언 시간(%), 발언/경청 비, 최장 독백(한 사람이 끊김 없이 가장 길게 말한 구간), 단어 수·분당 단어(WPM). 막대/도넛 차트 한 장.

**왜 필요한가.** Fathom의 "스코어카드", Read.ai의 "코치", Fireflies·tl;dv·Avoma가 모두 2026에 **핵심 차별점으로 미는** 영역이다 — 시장이 이쪽으로 가고 있다. 회의 코칭·균형 잡힌 발언·"내가 너무 말이 많았나"는 팀 회의에서 즉각 와닿는 가치다.

**지금 가진 것 (실측).** 계산에 필요한 원천 데이터가 **100% 갖춰져 있다**: `transcripts` 테이블에 `speaker_label`·`started_at_ms`·`ended_at_ms`·`content`·`speaker_name`이 전부 존재(`schema.rb` 340–353). 화자분리·이름까지 이미 들어가 있다. 프론트 `TranscriptPanel.tsx`도 있다. **DB 변경 0.**

**새로 만들 것.** 백엔드 `MeetingAnalyticsService` — `speaker_label`로 그룹화해 Σ(ended−started)=발언 시간, `content` 단어 수, WPM, talk/listen 비, 최장 독백을 산출. `GET /meetings/:id/analytics` 엔드포인트. 프론트 막대/도넛 차트 컴포넌트. **전부 순수 후처리**라 LLM·클라우드 비용 0.

**위험·제약.** confidence·단어 단위(per-word) 타이밍이 없어, 속도 변동성·감정 같은 정밀 지표는 불가. 세그먼트 단위 집계까지가 한계 — 그게 가치의 대부분이다.

**추천.** **가장 먼저 하라.** 가성비 1위. 신규 데이터·모델 없이 "시장이 가는 방향"을 가장 싸게 따라잡는다.

### ③ 액션아이템 제품 표면 (담당자·마감·"내 미결 액션") ★★★ · 공수 S~M

**무엇인가.** 회의에서 뽑힌 할 일에 **담당자와 마감일**이 실제로 붙고, "내가 맡은, 아직 안 끝난 일"을 여러 회의에 걸쳐 한 화면에서 본다. 마감 임박 알림까지.

**왜 필요한가.** Otter·Avoma·Sembly에서 **표준(table-stakes)**이다. 담당자·마감·태스크툴/CRM 푸시가 기본. 또박또박은 *데이터 모델이 제품보다 앞서 있는* 드문 경우 — 거의 다 만들어 놓고 마지막 배선만 안 했다.

**지금 가진 것 (실측).** 거의 완비다. `action_items` 테이블에 `assignee_id`·`due_date`·`status`·`ai_generated` 컬럼 존재(상태는 todo/in_progress/done 체크 제약). `ActionItem` 모델에 `belongs_to :assignee`. 프론트 `ActionItemForm.tsx`에 **담당자 드롭다운 + 마감일 입력이 이미 있다**. 직렬화도 assignee·due_date를 내려준다. **LLM 프롬프트는 이미 `assignee_hint`·`due_date_hint`를 추출하라고 지시**하고 있다.

**새로 만들 것 (핵심 = 끊긴 배선 잇기).** `MeetingFinalizerService#save_action_items`가 지금은 `content`/`status`/`ai_generated`만 저장하고 **LLM이 뽑은 `assignee_hint`·`due_date_hint`를 통째로 버린다**. 여기서 이름 퍼지 매칭(→ `User.id`)과 상대 일자 파싱(예: "다음 주 화요일" → date)을 해서 저장만 하면 된다. 추가로 `GET /action_items/my_open`(현재 사용자·미완료·교차회의), `config/recurring.yml`에 마감 임박 알림 크론.

**위험·제약.** 이름 퍼지 매칭은 동명이인·별칭에서 틀릴 수 있음 → "추정 담당자"로 표시하고 사용자가 확정하게 하면 안전. 태스크툴/CRM 푸시는 후속(⑫ 웹훅과 묶는 게 자연스럽다).

**추천.** **두 번째로 하라.** "이미 90% 만들어 둔 기능을 켜는" 작업이라 가성비가 비정상적으로 높다.

### ④ 즉석 다포맷 재구성 + 한국어 템플릿 ★★☆ · 공수 M

**무엇인가.** 같은 회의록을 버튼 한 번으로 **임원 보고용 / 화자별 / 한 줄 요약**으로 재성형하고, 주간·1on1·킥오프·인터뷰 같은 **기성 템플릿**을 제공.

**왜 필요한가.** Fathom·Teams·Granola·클로바노트가 커스텀 요약 지시문 + 기성 템플릿을 민다. "회의는 한 번, 출력은 여러 용도"는 보고가 잦은 한국 팀에 잘 맞는다.

**지금 가진 것 (실측).** 상세도·재구성 인프라가 있다. `meetings`에 `summary_verbosity`(기본 standard)·`summary_restructure`(기본 true) 컬럼. `LlmService#refine_notes`가 sections_prompt·verbosity 인자로 회의록을 재성형. `PromptTemplate` 모델의 `sections_prompt_for(meeting_type)` + `config.yaml`에 회의 **유형**별 섹션 9종(general/team/standup/brainstorm/review/interview/workshop/one_on_one/lecture).

**새로 만들 것.** 핵심 차이: 현재 있는 건 회의 *유형*별 "섹션 구조"이지, *출력 포맷* 변환이 아니다. 그리고 `refine_notes`는 **전사 기반 재-LLM**(비싸다). 추가할 것 = `LlmService#reconstruct_notes(기존 요약, 포맷)` — **전사 없이 이미 만든 요약만 저토큰으로 재성형**. 포맷 상수(임원보고/화자별/한줄/주간/1on1/킥오프/인터뷰), `POST /meetings/:id/summary/reconstruct?format=`.

**위험·제약.** 기존 요약이 빈약하면 재성형 품질도 그만큼만 나옴(쓰레기 입력→쓰레기 출력). "전사 기반 전체 재생성"과 "요약 재성형"을 UI에서 명확히 구분해야 사용자가 혼동 안 한다.

**추천.** Tier 1 셋 중 가장 큰 작업이지만 여전히 인프라 위 얇은 레이어. ①③ 다음.

## 4. Tier 2 — 전략적 중간 (LLM 레이어/모델·통합 추가)

### ⑨ 라이브 in-meeting 어시스턴트 (catch me up) ★★★ · 공수 M

**무엇인가.** 회의가 *진행되는 중에* "내가 방금 놓친 것 / 지금까지 나온 액션 / 내 이름이 언급됐나"를 실시간으로 알려 준다. 늦게 들어온 사람을 위한 "따라잡기".

**왜 필요한가.** MS Teams Copilot이 밀어 **enterprise table-stakes**가 된 영역(조사에서 3-0 확정). 두 조사 모두 또박또박의 **최대 미사용 자산**으로 지목했다. 가장 어려운 전제(라이브 페이지·실시간 채널·per-user 챗 RAG)를 이미 다 깔아 놨다.

**지금 가진 것 (실측).** 재사용 인프라가 §설계대로 전부 있다: `SummarizationJob`이 1분 크론으로 녹음 중 회의의 실시간 요약을 이미 돌린다. `ChatChannel`이 per-user 스트림(`meeting_{id}_chat_{user_id}`)을 권한 판정과 함께 제공. `MeetingChatContext`가 전사 RAG(+요약+히스토리). 즉 **"수동" 따라잡기(라이브 요약 읽고 챗으로 질문)는 사실상 이미 가능**.

**새로 만들 것.** 빠진 건 **능동(proactive) 푸시**다. `MeetingAssistantInsightsJob`(요약 크론 뒤 몇 분마다 가벼운 LLM) — "새 액션/결정/내 이름 언급"을 추출해 해당 사용자 채널에 `type:"assistant_insights"`로 브로드캐스트. 프론트는 `RightTabsPanel`에 인사이트 탭 하나 추가.

**위험·제약.** 라이브 LLM 호출 빈도 = 비용·부하. 너무 자주면 시끄럽고 비싸다 → 간격·"중요한 것만" 임계값을 보수적으로. 알림 피로 주의.

**추천.** **전략적 1순위.** "이미 깔린 가장 비싼 인프라를 드디어 쓰는" 작업. 라이브 가치라는 최대 빈틈을 정면으로 메운다.

### ⑫ 아웃바운드 웹훅 + 스코프드 API 토큰 ★★☆ · 공수 L

**무엇인가.** 회의 완료/요약 준비 시 외부로 **웹훅**을 쏘고(요약·액션아이템 페이로드), 팀이 **API 토큰(PAT)**으로 자기네 사내 메신저·Jira·노션에 직접 연동.

**왜 필요한가.** Otter·Avoma·Fireflies가 전부 outbound webhook + Bearer 토큰을 제공 → 이제 **선택이 아니라 경쟁 결손**으로 격상됐다. 그리고 이건 **자체호스팅에 가장 잘 맞는 통합** — 클라우드 SaaS가 막아 둔 걸 우리는 자유롭게 열 수 있다(차별점 강화).

**지금 가진 것 (실측).** 재사용 패턴만: `JwtService`(단 이건 Devise 로그인 토큰이지 API 권한 스코프 아님), `MeetingFinalizerService`(enqueue 지점), `User.llm_api_key` 암호화 컬럼 패턴, `SidecarClient`(Net::HTTP+에러처리). **PAT 개념은 전무** — `ApiToken` 모델·`token_digest`/`scopes`/`expires` 컬럼·Bearer 미들웨어·공개 REST 전부 없음.

**새로 만들 것.** (웹훅) `ApiWebhook` 모델(url·secret·이벤트 필터), `WebhookService#deliver`(HMAC-SHA256 서명 + 재시도), finalizer에서 `WebhookDeliveryJob` enqueue, CRUD UI. (API 토큰) `ApiToken` 모델, Bearer 미들웨어, `meetings:read` 류 스코프, 작은 공개 REST.

**위험·제약.** **보안 표면이 새로 열린다** — 토큰 스코프·만료·폐기, 웹훅 서명 검증, SSRF(웹훅 URL) 방어를 처음부터 제대로. effort L인 이유. 작게(읽기 전용 스코프 + 완료 웹훅 하나)부터 시작 권장.

**추천.** 자체호스팅 정체성에 **전략적으로 중요**하지만 보안 작업이라 서두르지 말 것. ③ 액션아이템과 묶으면(액션 푸시) 시너지.

### ⑱ 실시간 발화 코치 ★☆☆ · 공수 M — ⚠️데이터 제약

**무엇인가.** 말 속도, 필러워드("음/어"), 감정/참여를 실시간 코칭.

**제약이 본질.** 현 `transcripts`에 **단어 단위 타이밍·confidence가 없다** → 필러워드 정밀 탐지·속도 변동성 불가. MVP는 세그먼트 속도(duration÷words) + 패턴 매칭("음/어") 수준이 한계. 정밀화하려면 사이드카 word-timestamp가 필요(qwen3에 word-ts 없어 폐기한 이력). **추천:** ①과 데이터를 공유하니 ① 한 뒤 "가능한 만큼"만. 정밀 코치는 STT 모델 교체 전제 → 후순위.

---

### ⑲ 라이브 캡션 + 접근성(a11y) ★☆☆ · 공수 M

**무엇인가.** 회의 중 대형 고대비 실시간 캡션 + 스크린리더 지원(ARIA 라이브 리전).

**왜.** 한국 공공/교육 조달의 접근성 요건. 추천 목록에 0이던 빈틈. **지금 가진 것:** 라이브 전사는 이미 실시간 렌더(`transcriptStore.finals`). **새로 만들 것:** `LiveCaptionsPanel`(현재 ms 세그먼트, 대형 고대비) + `aria-live="polite"`, 화자 라벨 aria. SRT/VTT 동반 가능. **추천:** ①(분석)과 별개. 조달·접근성이 목표일 때.

## 6. 짧은 목록 — 낮은 우선순위 / 데이터 제약 (깊이 투자 비권장)

이 항목들은 §10 구현 설계가 없거나(애초 미설계), 데이터 제약이 본질이라 짧게만 둔다.

- **사전 회의 브리프** (none) — 회의 시작 전 "이 폴더서 지난번 X·미결 Y개"를 자동 브리프. 재료(이전 회의 참고·안건 첨부)는 있으나 파이프라인 전무. 폴더 챗(#1)·이전 회의 참고와 묶으면 의미. *공수 M, 가치 med.*
- **폴더 정기 AI 다이제스트** (none) — 폴더/프로젝트 단위 주기 요약 발송. `#1` 폴더 챗에 의존. 정기 잡·크론·발송 경로를 새로. *공수 M, 가치 med.* ⑧ 이메일·⑬ 에이전트와 묶임.
- **감정/참여 신호** (none) — ⑱과 같은 데이터 제약(confidence·word 타이밍 없음)이 근본 한계. EU-AI-Act 리스크도 있어 자체호스팅 스코핑 보수적으로. *깊이 투자 비권장.*
- **노트 버전 이력** (none) — 요약 재생성 시 기존 레코드를 덮어써 이력이 없다(휴지통 복구만 존재). PaperTrail류 도입 필요. ⚠️*문서화된 Ctrl+Z 데이터 손실(메모리: `summary_undo_dataloss`)과 직결* → 데이터 안전 관점에서 **가치는 낮지 않다.** 안전성 작업으로 재평가 권장. *공수 M.*
- **녹음 동의 캡처 + egress 제어** (none) — 동의 플로우·외부 전송 제어 전무. EU-AI-Act/동의 대응. 규제 진입 시. *공수 L.*
- **시각 렌더 확장(마인드맵·타임라인)** (partial) — `mermaidBlock`이 **타입 무제한 렌더**라 mindmap/timeline mermaid 문법이 들어오면 *이미 렌더된다.* 빠진 건 전용 UI와 "LLM이 그 문법을 산출하게 하는 프롬프트 가이드"뿐. *공수 S, 가성비 의외로 좋음* — 프롬프트만 손보면 거의 공짜.
