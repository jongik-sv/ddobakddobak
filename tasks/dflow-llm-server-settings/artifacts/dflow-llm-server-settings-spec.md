# D'Flow 서버 LLM 설정 — 프리셋 프로필 관리 이식 스펙

> **이 문서만 읽고 D'Flow(wbs-web) 저장소에서 바로 구현할 수 있도록 작성된 자기완결 스펙이다.**
> 원본(또박또박)의 "서버 LLM 설정 + 프리셋 프로필 관리" 기능을 D'Flow 스택(Next.js 15 App Router + React 19 + Supabase + Tailwind 4 + vitest)으로 번역했다. 원본 저장소 접근은 불필요하다.
> 대상 저장소: `~/project/wbs-web`

---

## 1. 목적·스코프

관리자(pmo_admin)가 **LLM 접속 정보를 "프로필"로 여러 개 등록해 두고, 그중 하나를 서버 전역 LLM으로 선택**할 수 있게 한다. 현재 D'Flow는 LLM 설정이 env 변수(`AI_PROVIDER`, `GEMINI_API_KEY`, `LLM_API_KEY` 등)에 하드코딩되어 있어 변경 시 재배포가 필요하다 — 이를 관리 화면에서 즉시 전환 가능하게 만든다.

**포함**:
- LLM 프로필 CRUD (관리자 전용)
- 전역 활성 LLM 선택: `환경변수 기본값` / `프로필 선택` / `선택 안함`
- 연결 테스트 (프로필 폼 안 + 활성 선택 화면)
- 기존 `lib/ai/provider.ts` 소비 코드와의 통합 (콜사이트 무수정)

**제외** (원본에는 있으나 D'Flow에 불필요):
- 사용자별 개인 LLM 설정/개인 프로필 풀 — D'Flow는 서버 전역 설정만 사용
- CLI 프리셋(claude_cli 등) — D'Flow에 CLI 실행 인프라 없음
- 임베딩 설정(`embedConfig()`, `EMBED_*` env) — **건드리지 않는다.** 이번 범위는 텍스트 생성 LLM(`llmConfig()`)만
- AI 챗용 별도 프로필 선택 — D'Flow의 모든 LLM 소비처는 단일 전역 설정을 공유

## 2. 개념 모델

| 용어 | 정의 |
|---|---|
| **LLM 프로필** | 이름 붙은 LLM 접속 정보 한 벌: provider, base_url, model, auth_token, 토큰 한도. 여러 개 등록 가능 |
| **프리셋(preset)** | 프로필 생성 폼에서 provider/base_url을 자동으로 채워주는 서비스 템플릿 (Gemini, OpenAI, Ollama, LM Studio, 직접 입력). 프로필의 `preset_id`로 어느 템플릿에서 만들어졌는지 기록 |
| **전역 LLM 설정(mode)** | 서버가 어떤 LLM을 쓸지에 대한 싱글톤 상태. `env`(기존 env 변수 그대로 — 기본값) / `profile`(활성 프로필 사용) / `none`(LLM 기능 차단) |

프로필 관리(CRUD)와 활성 선택은 **분리된 두 상태**다. 프로필을 만들어도 활성으로 선택·저장하기 전에는 아무 효과가 없다.

### 프리셋 정의 (프런트 상수)

| preset_id | 라벨 | provider | base_url 기본값 | auth_token |
|---|---|---|---|---|
| `gemini` | Google Gemini | `gemini` | (빈값 — 구글 기본 엔드포인트) | 필요 |
| `openai` | OpenAI | `openai` | (빈값 — api.openai.com) | 필요 |
| `ollama` | Ollama | `openai` | `http://localhost:11434/v1` | 불필요 |
| `lmstudio` | LM Studio | `openai` | `http://localhost:1234/v1` | 불필요 |
| `custom` | 직접 입력 (OpenAI 호환) | `openai` | (빈값 — 사용자 입력) | 선택 |

- provider는 D'Flow의 기존 `AiProvider = 'gemini' | 'openai'` 2종만. OpenAI 호환 서버(Ollama/LM Studio/Groq/사내 LLM)는 전부 `openai` + `base_url`로 커버한다 (기존 `LLM_BASE_URL` 관례와 동일).
- "auth_token 불필요" 프리셋은 키 없이 저장·테스트 가능해야 한다 (loopback/사내 서버).

## 3. 데이터 모델 — `supabase/migrations/0038_llm_config.sql`

> 파일 번호는 작성 시점 기준 최신이 `0037_wbs_embeddings_doc_key.sql`이라는 전제다. **구현 시 `ls supabase/migrations/`로 최신 번호를 확인해 +1로 조정하라.** 롤백 파일 `0038_llm_config_rollback.sql`도 페어로 작성한다(저장소 관례, `0028_*` 참조). 파일 헤더에 목적 1줄 + 적용 방법 주석(기존 파일 관례 모방).

```sql
-- LLM 프로필 + 전역 LLM 설정 (관리 화면에서 서버 LLM 전환)
create table if not exists llm_profiles (
  id bigint generated by default as identity primary key,  -- 저장소 선례(0031) 관례
  name text not null unique,
  preset_id text not null,
  provider text not null check (provider in ('gemini', 'openai')),
  base_url text,
  model text not null,
  auth_token text,
  max_input_tokens int,
  max_output_tokens int,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists llm_config (
  id int primary key default 1 check (id = 1),  -- 싱글톤
  mode text not null default 'env' check (mode in ('env', 'profile', 'none')),
  active_profile_id bigint references llm_profiles(id) on delete set null,
  updated_by uuid references auth.users(id),
  updated_at timestamptz not null default now()
);
insert into llm_config (id) values (1) on conflict do nothing;

alter table llm_profiles enable row level security;
alter table llm_config enable row level security;

-- 두 테이블 모두 pmo_admin 전용 (auth_token이 있으므로 읽기도 잠근다).
-- 런타임 소비는 service_role 캐시 로더가 담당하므로 일반 사용자 select 불필요.
drop policy if exists admin_all_llm_profiles on llm_profiles;
create policy admin_all_llm_profiles on llm_profiles for all to authenticated
  using (app_role() = 'pmo_admin') with check (app_role() = 'pmo_admin');

drop policy if exists admin_all_llm_config on llm_config;
create policy admin_all_llm_config on llm_config for all to authenticated
  using (app_role() = 'pmo_admin') with check (app_role() = 'pmo_admin');
```

설계 근거 (원본에서 검증된 규칙의 번역):
- **활성 프로필 삭제 시 참조 해제**: 원본은 모델 콜백으로 nullify했다. D'Flow는 Postgres이므로 `on delete set null` FK로 해결 (해석 규칙은 §5).
- **이름 유니크**: 단일 풀이므로 단순 `unique`. 중복 시 사용자 메시지는 §7.
- `app_role()`은 저장소에 이미 존재하는 헬퍼 함수다(`0012_announcements.sql`에서 생성, memberships.role 조회). 재정의하지 말 것.
- `auth_token`은 평문 저장 + RLS 잠금이다(내부 도구 전제, 원본도 동급). 강화가 필요해지면 Supabase Vault로 후속 전환 — 이번 범위 아님.

## 4. 서버 액션 계약 — `src/app/actions/llmConfig.ts`

Route Handler가 아니라 **Server Action**으로 구현한다(저장소 1차 관례, `src/app/actions/accounts.ts` 참조). 파일 최상단 `'use server'`.

**공통 게이트**: 모든 액션 최상단에서 관리자 확인. `accounts.ts`의 `isAdmin()`은 **export되지 않은 모듈 로컬 함수**이므로 import하지 말고, 동일 로직(`getMembership()` 후 `role !== 'pmo_admin'`이면 `{ error: '권한이 없습니다' }` 반환)을 `llmConfig.ts`에 복제한다. RLS가 이중 방어지만 액션 게이트를 생략하지 말 것.

**DB 클라이언트**: 액션 내부는 `createServerClient()`(RLS 경유)로 충분. `service_role`은 §5의 런타임 캐시 로더에서만 사용.

**마스킹 규칙** (모든 응답 공통): `auth_token` 원문은 **어떤 액션 응답에도 절대 포함하지 않는다.** 대신:
- `has_token: boolean`
- `auth_token_masked: string | null` — `token.slice(0,4) + '...' + token.slice(-4)`, 길이 8자 이하면 `'****'`, 토큰 없으면 `null`

```ts
export interface LlmProfileMasked {
  id: number
  name: string
  preset_id: string
  provider: 'gemini' | 'openai'
  base_url: string | null
  model: string
  max_input_tokens: number | null
  max_output_tokens: number | null
  has_token: boolean
  auth_token_masked: string | null
}

export interface LlmProfileInput {
  name: string
  preset_id: string
  provider: 'gemini' | 'openai'
  base_url?: string
  model: string
  auth_token?: string          // 빈 문자열/미전송 = "변경 없음" (아래 규칙)
  max_input_tokens?: number
  max_output_tokens?: number
}
```

| 액션 | 시그니처 | 동작 |
|---|---|---|
| `listLlmProfiles` | `() => { profiles: LlmProfileMasked[] } \| { error }` | name 순 정렬 |
| `createLlmProfile` | `(input: LlmProfileInput) => { profile } \| { error }` | 검증(§7) 후 insert. `auth_token` 빈값이면 null 저장 |
| `updateLlmProfile` | `(id: number, input: LlmProfileInput) => { profile } \| { error }` | **`auth_token`이 빈 문자열이거나 미전송이면 update 대상에서 그 컬럼을 제외해 기존 키를 유지한다** (원본 검증 규칙 — 빈값 재전송으로 키가 지워지는 사고 방지). `updated_at = now()` |
| `deleteLlmProfile` | `(id: number) => { ok: true } \| { error }` | 삭제. FK가 `llm_config.active_profile_id`를 자동 null 처리. 삭제 후 `refreshLlmOverride()` 호출(§5) |
| `getLlmConfig` | `() => { mode, active_profile_id, profiles: LlmProfileMasked[] } \| { error }` | 관리 화면 초기 로드용 원샷 |
| `saveLlmConfig` | `({ mode, active_profile_id }) => { ok: true } \| { error }` | `mode='profile'`인데 `active_profile_id`가 없거나 존재하지 않는 프로필이면 `{ error: '유효하지 않은 프로필입니다' }`. `mode='env'\|'none'`이면 `active_profile_id`는 null로 저장. `updated_by` 기록. 성공 시 **반드시 `await refreshLlmOverride()`**(§5 — 안 하면 TTL까지 옛 설정 사용) |
| `testLlmConnection` | `({ provider, model, base_url?, auth_token?, profile_id? }) => { success: boolean, error?: string }` | 실제 LLM 1회 호출(§6). **키 폴백 체인**: `auth_token`이 비어 있고 `profile_id`가 있으면 그 프로필의 저장된 키를 서버에서 조회해 사용 — 편집 중 키를 다시 입력하지 않아도 테스트 가능(원본 UX 규칙) |

mutating 액션 성공 시 `revalidatePath('/admin/llm-config')`.

## 5. AI 레이어 통합 — 이 이식의 핵심 난관

### 현황 (건드리기 전에 반드시 파악)

`src/lib/ai/provider.ts`의 `llmConfig()` / `hasLLM()` / `aiProvider()`는 **동기 함수**이고, `lib/ai/` 하위 8개 파일(`llm.ts`, `brief.ts`, `risk-brief.ts`, `minutes-insights.ts`, `answer.ts`, `minutes-answer.ts`, `health.ts` 등 — 정확한 목록은 `grep -rl "llmConfig()\|hasLLM()" src/`로 실측하라; `retrieve.ts`/`ingest.ts`는 임베딩 함수만 쓰므로 무관)에서 동기 호출된다. DB에서 설정을 읽도록 이 함수들을 async로 바꾸면 콜사이트 전체에 await 전파가 일어난다 — **하지 말 것.**

추가로 주의: `lib/ai/llm.ts`의 생성 함수들(`generateAnswer`, `generateAnswerStream`)은 `hasLLM()`과 **별개로** 내부에서 `if (!cfg.apiKey) return null` 가드를 갖는다. 이 이중 가드가 아래 apiKey 해석 규칙의 근거다.

### 채택 설계: 인메모리 캐시 + 동기 시그니처 유지

신규 파일 `src/lib/ai/llm-override.ts` (첫 줄 `import 'server-only'`):

```ts
interface LlmOverride {
  mode: 'env' | 'profile' | 'none'
  // baseUrl은 로드 시점에 항상 채워진다(아래 해석 규칙) — undefined 금지(strict 타입·URL 조합 안전)
  profile: { provider: 'gemini' | 'openai'; apiKey: string; baseUrl: string; model: string } | null
}

let cache: { value: LlmOverride; fetchedAt: number }   // 모듈 초기화가 반드시 채움 — null 상태 없음
let inflight: Promise<void> | null = null
const TTL_MS = 60_000

// 모듈 초기화(top-level await): 최초 로드를 1회 await한다. 3초 타임아웃,
// 실패 시 { mode: 'env', profile: null } + 서버 로그 1줄 후 기동 계속.
// 이유: lazy 로드면 콜드스타트 인스턴스가 '선택 안함(none)' 저장 상태를 모른 채
// env 키로 LLM을 호출하는 fail-open이 생긴다. 초기화 await가 이를 차단한다.

// 동기: 캐시 반환(항상 존재). TTL 만료면 백그라운드 갱신을 fire-and-forget으로
// 트리거하고 현재 값을 즉시 반환. 서버리스에서 백그라운드 프라미스가 죽어도
// 다음 요청이 다시 트리거하므로 안전(stale은 직전 유효 설정일 뿐 fail-open 아님).
export function llmOverrideSync(): LlmOverride

// 비동기: DB 즉시 재조회 + 캐시 교체. saveLlmConfig/deleteLlmProfile에서 await.
export async function refreshLlmOverride(): Promise<void>
```

- DB 조회는 **`createAdminClient()`(service_role, `src/lib/supabase/admin.ts`)** 사용 — RLS가 admin 전용이므로 런타임(일반 사용자 요청 중의 LLM 호출)에서는 service_role로만 토큰을 읽을 수 있다. `server-only` 표시로 클라이언트 번들 유입 차단.
- 조회 쿼리: `llm_config` 1행 + (mode='profile'이면) 활성 프로필 조회(2쿼리로 단순 구현 가능 — join 불필요).
- **로드 시 해석(resolve) 규칙 — 값 정규화를 캐시에 넣기 전에 끝낸다**:
  1. `mode='profile'`인데 `active_profile_id`가 null(프로필 삭제됨) → `mode:'env'`로 해석 (dangling 안전 폴백 — 원본 검증 규칙)
  2. `baseUrl`: 프로필 `base_url`이 비어 있으면 provider 기본 엔드포인트로 채운다 — gemini `https://generativelanguage.googleapis.com/v1beta`, openai `https://api.openai.com/v1`. **이 값들은 이미 `provider.ts`의 기존 env 폴백에 하드코딩되어 있으므로, 공유 상수로 추출해 양쪽이 같은 값을 쓰게 하라** (빈 base_url 프로필 = gemini/openai 프리셋의 정상 케이스이므로 이 해석이 없으면 `${cfg.baseUrl}/...` URL이 깨진다)
  3. `apiKey`: `auth_token`이 있으면 그 값. 없으면 — `provider='openai'`이면 placeholder 문자열 `'local'` (Ollama/LM Studio 등 키 불필요 서버는 Authorization 헤더 값을 무시하므로 무해. 빈 문자열이면 `llm.ts`의 `if (!cfg.apiKey) return null` 가드에 걸려 실사용이 조용히 무동작이 된다). `provider='gemini'`는 키 필수 — 키 없으면 빈 문자열 유지(아래 `hasLLM()` 규칙이 미구성으로 처리)
- 동시 갱신 방지: `inflight` promise 재사용.

### `provider.ts` 수정 (기존 함수 시그니처 불변)

```ts
// llmConfig() 내부 (개념 — 실제 코드는 기존 스타일에 맞출 것)
const ov = llmOverrideSync()
if (ov?.mode === 'profile' && ov.profile) {
  return { provider: ov.profile.provider, apiKey: ov.profile.apiKey,
           baseUrl: ov.profile.baseUrl, model: ov.profile.model /* 기존 반환 형태에 맞춤 */ }
}
if (ov?.mode === 'none') return /* 키 없음과 동일한 미구성 상태 */
// ov가 null이거나 mode='env' → 기존 env 로직 그대로
```

- `hasLLM()`도 동일 분기: `mode='none'`이면 `false`. `mode='profile'`이면 — openai는 항상 `true`(apiKey는 로드 시 placeholder까지 보장됨), gemini는 `apiKey`가 실제로 있어야 `true`. 그 외 기존 로직.
- **`mode='none'`의 의미**: LLM 기능 전면 차단. D'Flow는 이미 `hasLLM()===false`일 때의 폴백 경로가 모든 소비처에 구현돼 있으므로, none을 "미구성"과 동일하게 취급하면 추가 차단 코드가 필요 없다. (원본은 이를 위해 별도 에러 타입을 도입했지만 D'Flow는 기존 폴백 구조 덕에 불필요.)
- `embedConfig()` / `hasEmbeddings()`는 **수정 금지** (임베딩은 env 유지).
- 알려진 트레이드오프(스펙 승인 사항): ①모듈 초기화 로드가 실패하면(DB 순단) `mode:'env'`로 기동한다 — 가용성 우선 결정. '선택 안함'이 보안 요구가 되는 시점이 오면 이 폴백만 `none`으로 바꾸면 된다. ②멀티 인스턴스 배포에서 관리 화면 저장은 해당 인스턴스에만 즉시 반영되고 다른 인스턴스는 최대 TTL(60초) 후 반영.

### 연결 테스트 구현 (§4 `testLlmConnection`)

- `provider='gemini'`: `generateContent` REST 1회, 프롬프트 `"ping"`. **base_url 반영**(비어 있으면 §5 해석 규칙과 동일하게 기본 엔드포인트) — 실사용 코드(`llm.ts`)가 `${cfg.baseUrl}/models/...`로 조합하므로 테스트도 같은 URL 규칙이어야 실사용과 어긋나지 않는다.
- `provider='openai'`: `chat.completions` 호환 fetch 1회, base_url 반영 동일.
- **재시도 금지**: `lib/ai/util.ts`의 `fetchWithRetry`는 재시도+429 백오프가 있어 테스트 UX(빠른 실패 표시)와 안 맞는다. 단발 `fetch` + `AbortController` 10초 타임아웃으로 별도 구현.
- 성공 → `{ success: true }`, 실패 → `{ success: false, error: <HTTP 상태·응답 메시지 요약> }`. **에러 메시지에 auth_token을 포함시키지 말 것.**

### 화면 1 — 활성 LLM 선택 카드 (LlmConfigManager)

- 선택지 3개 (라디오 또는 세그먼트 버튼, 저장 전까지 로컬 상태):
  1. **환경변수 기본값** — "배포 env 설정(AI_PROVIDER 등)을 그대로 사용합니다"
  2. **프로필 선택** — 드롭다운(등록된 프로필 목록: `이름 — 모델`) + 마지막 항목 "＋ 새 프로필 만들기…"(선택 시 값 변경 없이 프로필 관리 모달을 생성 폼 상태로 오픈)
  3. **선택 안함** — "LLM 기능(요약·브리프·답변 등)을 실행하지 않습니다"
- 카드 헤더에 **"프로필 관리"** 버튼 → 모달 오픈(목록 상태).
- **연결 테스트** 버튼: 현재 선택 상태로 즉시 테스트(저장 불필요). '프로필 선택'이면 `profile_id` 동봉(저장된 키 폴백). **'선택 안함' 상태면 disabled** (원본 규칙). '환경변수 기본값'이면 env 값이 서버에 있으므로 provider/model만 보내 테스트하거나 버튼 숨김 — 구현 단순화를 위해 env 모드에서는 버튼 숨김을 기본으로 한다.
- **저장** 버튼 → `saveLlmConfig`. 성공 토스트(`components/ui/Toast.tsx`), 실패 시 에러 문자열 표시.
- **dangling 폴백**: 모달에서 CRUD 후 목록이 갱신됐을 때, 현재 드롭다운이 가리키는 프로필이 삭제로 사라졌으면 선택 상태를 '환경변수 기본값'으로 자동 전환한다(저장 시 "유효하지 않은 프로필입니다" 422성 오류 방지 — 원본 검증 규칙).

### 화면 2 — 프로필 관리 모달 (LlmProfilesModal)

- **목록 상태**: 행마다 `이름 · 프리셋 라벨 · 모델 · auth_token_masked` + [편집] [삭제]. 상단 [＋ 새 프로필].
- **삭제**: `confirm()` 문구 — `'{name}' 프로필을 삭제할까요? 이 프로필을 쓰는 설정은 해제됩니다.` 확인 시 `deleteLlmProfile` → 목록 리로드 → 부모에 변경 통지(dangling 폴백 트리거).
- **생성/편집 폼** (목록 ↔ 폼 전환, 모달 안):
  - **프리셋 그리드**: §2의 5종 버튼. 클릭 시 provider/base_url을 프리셋 값으로 채움(이미 입력된 이름·키는 유지). 편집 시 초기 선택은 저장된 `preset_id`.
  - 필드: 이름(필수) / base_url / 모델(필수, 텍스트 입력) / API 키 / 최대 입력 토큰 / 최대 출력 토큰(숫자, 선택).
  - **API 키 입력 UX** (원본 규칙 그대로):
    - `type="password"` + 표시/숨김 토글(lucide `Eye`/`EyeOff`)
    - 편집 시 placeholder에 `auth_token_masked` 노출, 아래 안내문 `현재: {masked} — 비워두면 기존 키 유지`
    - 저장 payload에서 키가 빈값이면 **필드 자체를 생략**(§4 유지 규칙과 페어)
  - **폼 내 연결 테스트** 버튼: `disabled = !model`. 키 입력 중이면 그 키로, **빈칸이면 `profile_id`로 저장된 키 폴백**(신규 생성 중 + 키 빈칸 + 키 필요 프리셋이면 실패 응답을 그대로 표시).
  - 저장 → `createLlmProfile`/`updateLlmProfile` → 성공 시 목록 상태로 복귀 + 리로드.

스타일: 저장소 시맨틱 클래스(`bg-surface`, `border-line`, `text-ink-muted` 등) + 인라인 Tailwind 조합(전용 버튼/인풋 컴포넌트 없음이 관례). 아이콘 lucide-react.

## 7. 검증·에러 규칙

서버 액션에서 검증(클라이언트 검증은 보조). 실패 시 `{ error: string }`:

| 규칙 | 메시지 |
|---|---|
| `name` 필수·공백 불가 | `프로필 이름을 입력하세요` |
| `name` 중복 (unique violation, Postgres `23505`) | `이미 같은 이름의 프로필이 있습니다` |
| `model` 필수 | `모델을 입력하세요` |
| `provider`가 `gemini`/`openai` 외 | `유효하지 않은 provider입니다` |
| `preset_id` 필수 | (폼에서 항상 세팅되므로 서버는 존재만 확인) |
| `saveLlmConfig` mode='profile' + 프로필 부재 | `유효하지 않은 프로필입니다` |
| 비관리자 호출 | `권한이 없습니다` |

- `base_url` 형식 검증은 하지 않는다(원본도 없음 — 연결 테스트가 실질 검증).
- 키 필요 프리셋에서 키 없이 저장하는 것을 **막지 않는다**(원본 동일 — 저장은 자유, 연결 테스트/실사용에서 실패로 드러남).

## 10. 수용 기준 (수동 검증 시나리오)

1. 비관리자 로그인 → `/admin/llm-config` 접근 시 `/projects`로 리다이렉트, 헤더에 링크 미노출
2. 관리자: 프로필 생성(Gemini 프리셋, 키 입력) → 목록에 마스킹 키로 표시, 원문 미노출 (브라우저 devtools 응답 확인)
3. 폼에서 연결 테스트: 올바른 키 → 성공, 틀린 키 → 실패 메시지
4. 프로필 편집에서 키 빈칸 저장 → 기존 키 유지(연결 테스트 여전히 성공)
5. 활성 선택: '프로필 선택' + 해당 프로필 저장 → LLM 사용하는 기능(회의록 인사이트/브리프 등) 1회 실행 시 해당 프로필의 모델로 호출됨 (서버 로그의 모델명으로 확인)
6. '선택 안함' 저장 → LLM 기능이 미구성 폴백 경로로 동작(에러 아닌 안내/생략)
7. '환경변수 기본값' 저장 → 이식 전과 동일 동작 (회귀 없음)
8. 활성으로 선택된 프로필 삭제 → 화면 선택이 '환경변수 기본값'으로 폴백, 저장 오류 없음, 서버도 env로 동작
9. 이름 중복 생성 시도 → `이미 같은 이름의 프로필이 있습니다`
10. Ollama/LM Studio 프리셋: 키 없이 저장·테스트 가능(로컬 서버 없으면 연결 실패 메시지가 정상). 로컬 서버가 떠 있으면 **실사용 경로(요약/브리프 등)도 정상 호출**됨 — placeholder apiKey 덕에 `llm.ts`의 빈 키 가드에 걸리지 않는지 확인
11. '선택 안함' 저장 후 **서버 재시작** → LLM 기능 여전히 차단됨 (콜드스타트에서 env로 되돌아가는 fail-open 없음 — §5 초기화 await 검증)
