## 0. 결론 요약 (TL;DR)

1. **웹뷰는 필수다.** AI 요약 에디터(BlockNote)와 mermaid를 "지금 컴포넌트 그대로" 쓰려면 웹뷰 없이는 불가능하다. BlockNote는 ProseMirror(브라우저 DOM) 위에 있고, mermaid는 DOM 레이아웃·텍스트 측정에 의존한다. 이 사실이 기술 선택을 사실상 결정한다 — **앱 전체가 웹뷰인 Tauri에서는 이 요구가 "공짜"고, 다른 모든 스택에서는 "웹뷰 아일랜드 + 브릿지"라는 추가 비용이다.**

2. **추천 경로는 "빅뱅 재작성"이 아니라 "점진 이관으로 같은 목적지 도달"이다.** 목적지 = 현행 Tauri v2 + React UI(51K LOC)를 그대로 유지하고, Rails(12K LOC)를 Rust 인프로세스 코어로, Python sidecar(7.7K LOC)를 Rust FFI(whisper-rs / sherpa-onnx / speakrs 크레이트)로 흡수한 **단일 실행체**. 프론트가 이미 `localhost /api/v1` HTTP로만 통신하므로 strangler(교살자 무화과) 패턴의 이음새가 공짜로 존재하고, SQLite 파일·스키마를 신구가 공유해 데이터 마이그레이션 자체가 없다. Flutter/KMP/React Native/네이티브 2벌은 전부 기각 — 최대 자산인 UI 51K LOC를 태우고 얻는 것이 거의 없다.

3. **"하는 것이 유리한가"의 답은 조건부 예(Yes, if).**
   - **macOS 단일 실행체화는 명확히 유리하다.** 현행 데스크톱 앱은 사실 단일 앱이 아니라 3개 상주 프로세스(웹뷰 + Rails + Python)에 사용자 머신에 Homebrew/ruby/uv를 부트스트랩하는 소스배포형이다 — "단일 실행체"의 정반대 극단이며, 이 부채는 시간이 갈수록 부식한다.
   - **Android "로컬 단독" 부분은 수요 검증이 선행되어야 한다.** 폰 단독판은 기술적으로 가능하나 필연적으로 열화판이다(STT 소형모델·임베딩 드롭·LLM은 클라우드 키 필수). 이미 서버 접속형 Android + 폰 로컬 STT 경로가 동작 중이므로, "Mac이 꺼진 상태에서 폰 단독으로 얼마나 필요한가"를 2~4주 실사용으로 측정한 뒤 범위를 확정하는 것이 합리적이다.
   - **최대 리스크는 기술이 아니라 규율이다.** 이 저장소의 실개발 이력(최근 9일 50커밋, 주 단위 신기능)상 "이관 기간 기능 동결"이 가장 깨지기 쉬운 고리다. 동결을 못 지키면 어떤 옵션도 일정이 발산한다.

4. **현실적 공수: 1인 + AI 에이전트 기준 10~14개월**(macOS 완성 기준 8~11개월 + Android 확장). AI가 코드 번역은 깎아주지만 검증은 못 깎아준다(실측: AI 공저 코드 이슈 1.7배, 리뷰 시간 +91%). Rails를 은퇴 전까지 "정답 오라클"로 삼는 차등 테스트가 이 병목의 유일한 구조적 완충이며, 이는 점진 경로에서만 성립한다.

## 3. Q1 — 웹뷰가 필요한가: **그렇다**

### 판정 근거

- **BlockNote**: React + ProseMirror/TipTap 기반. TipTap 공식 답변이 결론을 대신한다 — *"ProseMirror is heavily reliant on the Browser DOM… basically locking us on the web"*. React Native 생태계의 표준 우회책(10tap-editor)조차 웹뷰 임베드 + 브릿지다. 현 코드의 커스텀 스펙(mermaidBlock 298L, citationInline 100L — ⟦t:ms/s:화자⟧ 인용배지)은 전부 이 위에 있다.
- **mermaid**: 렌더가 DOM의 `getBBox()`/텍스트 측정에 의존. headless JS 엔진(QuickJS 등)만으로는 안 되고(병목이 JS가 아니라 DOM 레이아웃), mermaid 팀은 "브라우저 없는 렌더" 요청을 *closed as not planned* 처리했다. 실무 표준(mermaid-cli, mermaid-isomorphic)도 결국 진짜 브라우저(puppeteer/Playwright)를 띄운다.
- **참고(신규 발견)**: 2025~26년 사이 mermaid의 네이티브 재구현이 등장했다 — Rust `merman`(mermaid 11.15 패리티 목표, Swift/Kotlin/Flutter FFI 바인딩), Rust `mmdr`(1.4k★), Swift `beautiful-mermaid-swift`(Craft 제작사, 6개 타입 한정). 즉 **"표시 전용"이라면 탈웹뷰가 이론상 가능**해졌으나, 전부 1년 미만 신생이라 충실도·버전 추격 리스크가 있고, **편집기(BlockNote)는 어차피 웹뷰 필수**라 실익이 없다.

### 시사점

"현행 컴포넌트 유지 + 웹뷰 필수"라는 제약은 **Tauri(앱 전체가 웹뷰)에서는 비용 0**, 다른 스택에서는 "미니 React 앱 + 양방향 브릿지(블록 JSON 동기화, `__ddobakSeek`/`__ddobakSpeakerAt` 전역 계약, 테마 동기화)를 상시 유지"라는 지속 비용이 된다. Notion조차 모바일 네이티브 전환에서 에디터만은 마지막까지 웹으로 남겼다 — 블록 에디터의 네이티브화는 자금력 있는 팀에게도 최고비용 항목이다.

## 4. Q2 — 방법 비교: 아키텍처 옵션 6종 평가

### 4.1 공통 전제 (모든 옵션에 해당하는 사실)

- **Tauri v2 사이드카(외부 바이너리)는 모바일 미지원 확정** (공식 shell 플러그인 지원표 + 추적 이슈 #9774 2년째 open + 메인테이너 확답). Rails/Python을 Android에서 별도 프로세스로 띄우는 길은 존재하지 않는다. → **어떤 옵션을 골라도 백엔드 재작성은 공통 비용**이다. 옵션 간 차이는 "UI를 살리느냐"뿐이다.
- **Android 온디바이스 한계는 스택과 무관하게 동일**: whisper.cpp는 tiny/base 양자화 배치만 실용(스트리밍은 1초 오디오에 5~7초, ANR), KURE급 임베딩 543MB+는 과중, LLM CLI는 Android W^X 정책(targetSdk 29+ 앱 홈 디렉토리 execve 금지)으로 구조적 불가. → **Android판은 어느 스택이든 "다운스케일 동반 앱"이다.**
- Rust 코어 빌딩블록은 macOS에서 사실상 완성 단계: rusqlite bundled(FTS5 내장), sqlite-vec(Android/iOS 사전컴파일 배포), apalis(tokio 인프로세스 잡큐), whisper-rs(Metal/CoreML — M2에서 large-v3-turbo 10분 오디오 63초 ≈ 9.5배 실시간), **sherpa-onnx 공식 Rust API**(STT+화자분리+VAD 단일 라이브러리, Android 데모 APK까지 배포; 서드파티 sherpa-rs가 "공식 바인딩이 생겼다"며 아카이브됨).

### 4.3 옵션별 요지

**A/F (통합 권고) — Tauri 단일앱을 목적지로 한 점진 이관.**
React UI 51K LOC와 에디터 스택(3.2K LOC)을 유일하게 100% 보존한다. 프론트가 이미 `localhost /api/v1` HTTP 116종으로만 백엔드와 통신하므로, Rust(axum) 인프로세스 서버가 동일 계약을 서빙하고 미이관 엔드포인트는 Rails로 프록시하면 **UI 무수정으로 엔드포인트 그룹 단위 이관**이 가능하다. SQLite 공유로 데이터 마이그레이션이 없고, Rails를 은퇴 전까지 차등 테스트 오라클로 쓸 수 있다(빅뱅은 이 오라클을 먼저 죽인다). 가장 어려운 조각인 Android 인프로세스 STT는 자체 프로토타입으로 이미 GREEN. 약점: Android 백그라운드는 Tauri 코어가 비어 있어 Kotlin 글루 자체 부담(단, 실사 결과 **RecordingForegroundService.kt가 이미 FGS+wake lock 완성 구현으로 존재** — "화면 끈 채 녹음"은 이미 넘은 벽이고, 남은 공백은 "앱 전환 시 웹뷰 오디오 전송 단절"에 대한 네이티브 캡처 이관이다). 공수 10~14개월.

**B. Flutter + Rust 코어.**
부품 성숙도만 보면 크로스스택 4종 중 1위(flutter_rust_bridge=Flutter Favorite, FGS 마이크 녹음 기성품, Canonical이 2026-05부터 데스크톱 리드 메인테이너, appflowy_editor라는 유일한 네이티브 블록 에디터 대안). 그러나 UI 34K LOC(비테스트)를 Dart로 전면 재작성해야 하고, 순수 TS 자산(citationMarkers, stt/ 20파일)까지 소실되며, UI에는 오라클이 없는 빅뱅이다. 어차피 해야 하는 Rust 코어 이관(~5개월분)은 A/F와 완전히 동일하므로, Flutter가 추가로 사주는 것은 사실상 "Android FGS 기성품 하나"뿐. **Tauri 경로의 Kotlin FGS 스파이크가 실패하거나 Android를 macOS와 대등한 1급 타깃으로 격상할 때만 재고.**

**C. Kotlin/Compose Multiplatform.**
Android는 4종 중 최상(코드가 곧 네이티브 Kotlin)이지만, **macOS가 네이티브가 아니라 JVM 데스크톱**(공식 안정성 표에 non-JVM macOS 항목 자체가 없음, jpackage로 JDK 동봉)이고, 웹뷰는 서드파티 KCEF(Chromium 동봉, 수백MB급) — 주 사용 플랫폼인 macOS가 현행 Tauri보다 무겁고 어색해지는 역설. UI·백엔드·플랫폼 셸 3층 동시 재작성으로 공수 최대급. 기각.

**D. 네이티브 2벌 + Rust 공유 코어.**
"단일 실행체"와 "Android 품질"의 천장은 최고지만 16~22개월 + 완성 후에도 기능 1개당 3중 구현(Rust+SwiftUI+Compose). 그 끝에 도달해도 앱의 심장(BlockNote 에디터)은 여전히 웹뷰 아일랜드로 남는다. 1인 체제에서 좌초 확률 최고. 앱스토어급 상용 제품화로 피벗할 때만 유효. 기각.

**E. React Native.**
유일한 차별 강점(BlockNote React 컴포넌트 재사용)은 Tauri가 "51K 전부, 공짜로" 얻는 것을 "에디터 3.2K만, 비싸게" 사는 구조. 결정적으로 react-native-macos가 코어 대비 4마이너(~8개월) 지연에 같은 Microsoft의 react-native-windows보다도 후순위 — macOS 1차 타깃을 그 위에 세울 수 없다. 기각.

## 5. Q3 — 재작성하는 것이 유리한가

### 5.1 고전 문헌의 판정 기준을 이 케이스에 적용

| 기준 (재작성이 유리한 조건) | 이 케이스 | 판정 |
|---|---|:---:|

**합계 ≈ 5.5~6.5 / 8 — 문헌 기준으로 이례적으로 재작성-우호적이다.** 단, 이 우호성은 전적으로 "**UI와 스키마를 유지한 좁은 백엔드 스왑을 점진적으로**"라는 조건부다. Joel Spolsky의 "절대 하지 마라"는 이 케이스에 정확히 들어맞지 않지만(전제인 '굴러가는 것을 자발적으로 전량 폐기'가 성립 안 함), Fowler의 "빅뱅은 대부분 불타 추락한다"는 그대로 적용된다.

### 5.2 재작성 회의론 (레드팀) — 반드시 직시할 4가지

1. **기회비용이 실측으로 크다.** 최근 9일간 50커밋(하루 ~5.5커밋; 6/28 하루에 export/import 전체가 설계→완성). 이 속도의 개발자에게 10~14개월은 "있었을 기능 수십 개"의 소멸이고, 재작성 종료 시점의 앱은 잘해야 지금과 동일 기능이다.
2. **기능 동결이 최대 리스크.** 전 옵션의 killer risk에 공통 등장. 이 저장소의 행동 이력상 8개월+ 동결은 비현실적일 수 있고, 동결 없는 strangler는 이중 구현으로 일정이 발산한다(+50% 각오).
3. **숨은 도메인 지식 소실은 가설이 아니라 이 저장소의 사고 이력이다.** 75개 마이그레이션에 축적된 엣지케이스, 그리고 실제 사고들 — 6/16 마이그레이션 FK cascade로 전사·회의록 전멸(회의록 13개 영구 미복구), LIKE ESCAPE, CLI 인코딩 크래시, Tempfile 수명, writer-lock 경합, WKWebView confirm 삭제사고. 재작성은 이 검증된 방어 코드를 백지에서 다시 쓰는 일이다. → 라인 단위 이관 검토 + 차등 테스트 없이는 재발한다.
4. **Android 로컬 단독의 실수요가 미검증.** 이미 서버 접속형 Android + 폰 로컬 STT가 동작한다. 남는 갭은 "Mac이 완전히 꺼진 상태에서 폰 단독으로 요약·검색·편집까지"인데, 이 빈도를 뒷받침할 데이터가 없다. **10개월을 청구하는 쪽에 입증 부담이 있다.**

한편 회의론자 스스로 인정한 것: **"아무것도 안 함"도 무비용이 아니다.** 소스배포형 부트스트랩은 툴체인 드리프트마다 파손 위험이 있고(운영 부비트랩 다수 실재), 3프로세스 + Ruby/Python 이중 런타임 유지비는 계속 든다. 순수 현상 유지는 장기적으로 부식한다 — 그래서 회의론자조차 최종적으로 F(점진 이관)를 현상 유지보다 우위에 뒀다.

### 5.3 타당성 종합 판정

> **macOS 단일 실행체화: 하는 것이 유리하다 (부채 청산 + 배포 단순화 + 목적지가 어차피 필요).**
> **Android 로컬 단독화: 조건부 — 2~4주 수요 검증 게이트를 통과할 때만.** 게이트 실패 시 Android 파트를 드롭하고 "macOS 단일 바이너리화"로 범위를 절반 이하로 줄이는 것이 정직한 경로다.
> **방식: 빅뱅 금지, strangler 필수.** 어떤 경우에도 B~E(UI 폐기 계열)는 이 목표·이 체제에서 정당화되지 않는다.

## 6. 권장 실행 계획

### M0. 선행 게이트 (2~4주, 이것부터)

착수 여부와 범위를 결정하는 4개 스파이크. **비용이 무시 가능하므로 10개월 커밋보다 순서상 반드시 앞선다.**

1. **Android 수요 실측**: 현행 서버 접속형 Android + 폰 STT로 실사용하며 "Mac 부재 상황"이 실제로 얼마나 발생하는지 기록.
2. **Android 앱 전환 내성 스파이크**: RecordingForegroundService.kt(이미 완성)를 토대로, 앱이 백그라운드로 간 상태에서 네이티브 캡처→업로드가 유지되는지 검증. (기존 인식 "FGS 전체 신작 필요"는 실사 결과 과장 — 진짜 공백은 웹뷰 오디오 경로의 네이티브 이관뿐.)
3. **macOS STT 품질 스파이크 — 실시간 한정** (배치는 기존 실측으로 종결): 자체 실측 `docs/stt-batch-engine-experiments-2026-06.md`(2026-06-13, 회의 152 = 47.9분 한국어 5화자)에 따르면 **배치 STT는 whisper.cpp gguf f16이 이미 품질·안정 1위**(세그먼트 657 최다, 스킵 0, 환각 없음, 10.3x; MLX greedy 계열은 환각으로 폐기 판정, Qwen3-ASR은 30초 뭉갬으로 배치 부적합). whisper-rs는 같은 엔진+같은 gguf 모델이므로 **배치 이식 = 무손실**(MLX beam 8bit 14.1x 대비 속도 ~27% 손해만). 남는 미지수는 **실시간 라이브 전사**(Apple Silicon 기본 = Qwen3-ASR 8bit MLX) — 스파이크는 §9의 인프로세스 후보 3종(transcribe.cpp Qwen3 GGUF · sherpa-onnx Qwen3 0.6B int8 · fluidaudio-rs CoreML)을 현행 MLX 기준선과 한국어 회의 오디오로 실측 비교하는 것으로 확정 — 선행 사례 조사(2026-07-04) 결과 "재구현" 없이 전부 기성 라이브러리 통합 수준이라 스파이크당 1~3일.
4. **기능 동결 자기 검증**: 2주간 신기능 0커밋을 실제로 지켜보기. 못 지키면 이관 단위를 더 잘게(그룹당 2주) 끊거나 범위를 축소.

### 부수 결정 사항

- **LLM 전략**: 데스크톱 = 현행 4티어 유지(CLI/BYOK/Ollama·MLX). Android = **BYOK 클라우드 기본**(회의 1시간 요약 $0.002~0.07 수준, 키는 Android Keystore) + 온디바이스(Gemma 3 4B급)는 오프라인 짧은 Q&A 폴백 한정. CLI는 Android에서 구조적 불가(W^X) — 데스크톱 전용으로 못박는다. (참고: Gemini CLI 무료 경로가 2026-06 종료된 사례처럼 CLI 우회는 정책 리스크도 크다.)
- **임베딩/의미검색**: mac은 KURE ONNX 포팅(중난도) 또는 유지, Android는 FTS-only로 시작. EmbeddingGemma(308M)는 한국어 품질이 KURE 대비 미검증이라 자체 벤치 전 채택 금지.
- **ffmpeg 5개 호출점**: Android는 바이너리 스폰 불가이므로 symphonia/rubato 계열 라이브러리 링크로 대체(M3~M4에서).

## 9. 추가 조사 (2026-07-04) — 실시간 STT 인프로세스 선행 사례

후속 질문 "남이 이미 인프로세스로 만들어놓은 것이 있는가"에 대한 웹 조사 4축(에이전트 4) 결과. **결론: 있다, 그것도 3벌 — "재구현"은 필요 없고 전부 라이브러리 통합 수준이다.** 배경: 현행 실시간 STT는 `settings.yaml` 기준 `qwen3_asr_8bit`(MLX, Python sidecar, Apple Silicon 전용), 배치는 `whisper_cpp`.

### 9.1 인프로세스 후보 서열

| 순위 | 경로 | 실체 | 강점 | 리스크 |
|---|---|---|---|---|
| **1** | **transcribe.cpp + Rust 바인딩** (`transcribe-cpp` 0.1.1, MIT) | Handy 팀의 ggml 기반 멀티모델 STT 엔진. **현행과 같은 Qwen3-ASR 0.6B/1.7B를 GGUF(Q8_0 811MB/2.08GB)로 인프로세스 실행, Metal/Vulkan/CUDA/CPU** — 즉 MLX 없이 같은 모델·GPU 가속. whisper 12종(large-v3-turbo 포함)도 같은 엔진 → **배치+실시간 의존성 단일화** 가능. 스트리밍 API(`stream.feed()` → committed/tentative + CommitPolicy)가 실시간 UX 완성형 | 품질 회귀 가능성 최소(동일 모델), 크로스플랫폼, 단일 의존성 | **프로젝트가 갓 공개**(86★, 바인딩 2026-07-03 게시). OnlineRecognizer·VAD·**화자분리**·화자임베딩까지 전면 래핑, prebuilt 정적 링크 자동 | ondevice-stt 실증 경로의 정식판(수제 FFI → 크레이트 대체), **Android와 같은 코드** | offline 전용(VAD+offline = 공식 권장 simulated streaming), 1.7B은 공식 변환본 없음(자가 export 스크립트 존재), **맥에서 CoreML EP가 CPU보다 느림**(이슈 #2910) → CPU 추론 |

### 9.2 훔칠 아키텍처 — Handy (Tauri v2, MIT, 25.6k★)

Tauri 인프로세스 실시간 STT의 완성 레퍼런스. 재사용 대상 패턴: cpal 녹음 → Silero VAD(SmoothedVad 프리필/행오버) → StreamRouter(오디오 콜백은 relaxed atomic 1회만, Feed/Finalize 동일 mpsc 채널 FIFO로 프레임 유실 방지) → 전용 워커 스레드 + RAII 가드 + `catch_unwind`. 우리 라이브 녹음의 락 경합·크래시 격리 문제와 같은 부류를 이미 푼 설계다. 반례 데이터도 하나: **vibe**(6.6k★ 전사 앱)는 v3.0.6에서 인프로세스를 버리고 whisper.cpp HTTP 사이드카(sona)로 회귀 — ML 크래시 격리 관점의 경고로, 인프로세스 채택 시 Handy식 `catch_unwind`+워커 격리가 필수라는 뜻.

### 9.3 시사점 — 계획 수정

1. **"MLX 대체 = 품질 도박"이라는 전제가 무너짐.** transcribe.cpp가 같은 Qwen3-ASR을 GGUF+Metal로 돌리므로, 잘 되면 품질 회귀 0에 Python 완전 제거. M0-3 스파이크는 "대안 모델 품질 검증"에서 "**동일 모델의 런타임 3종 비교**"(MLX 기준선 vs transcribe.cpp GGUF vs sherpa-onnx int8 [vs fluidaudio ANE])로 성격이 바뀌고, 각 1~3일이면 된다.
2. **화자분리도 인프로세스 선택지 확대**: speakrs 크레이트 직결(기존 계획) 외에 sherpa-onnx 화자분리(크로스플랫폼)와 fluidaudio LS-EEND(맥, 스트리밍)가 같은 크레이트에서 나옴.
3. **의존성 전략**: 이상적 조합 = transcribe.cpp(배치 whisper + 실시간 Qwen3 청크)로 단일화하되, 프로젝트가 어리므로 sherpa-onnx(공식, Android 공유)를 아키텍처상 교체 가능한 추상 뒤에 두는 이중 트랙. vibe 교훈대로 STT 호출은 워커 스레드 + panic 격리 필수.
